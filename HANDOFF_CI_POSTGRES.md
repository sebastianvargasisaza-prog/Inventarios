## Hallazgo 1 (CAUSA PRINCIPAL): DDL en dialecto SQLite no se traduce a Postgres
- Hay 261 sentencias `CREATE TABLE ... INTEGER PRIMARY KEY AUTOINCREMENT`.
  `AUTOINCREMENT` es SOLO de SQLite; Postgres necesita SERIAL / GENERATED AS IDENTITY.
- Existe el switch de dialecto `_usa_postgres()` (api/database.py:37, activado por
  EOS_DB_BACKEND=postgres) pero NO traduce el DDL.
- En una BD Postgres nueva, esos CREATE fallan y — por el patrón legacy
  `try: conn.execute(sql); except: pass` (api/database.py:~203-220) — el error se
  TRAGA en silencio, así que la tabla nunca se crea y luego revienta "relation
  does not exist". Ejemplos concretos:
    - producto_formula_alias  (api/database.py:486)
    - stock_por_entrar        (api/database.py:10091)
    - ventas_diarias          (api/database.py:10132)
- FIX (punto único, NO editar 261 statements): las migraciones se ejecutan en el
  loop `for stmt in stmts:` dentro de `run_migrations` (api/database.py:10199,
  loop en ~10257). Añadir ahí un shim de traducción que, si `_usa_postgres()`,
  reescriba las SQLite-ismos antes de ejecutar:
    * `INTEGER PRIMARY KEY AUTOINCREMENT` -> `GENERATED ALWAYS AS IDENTITY PRIMARY KEY`
      (o `SERIAL PRIMARY KEY`)
    * revisar otros: `AUTOINCREMENT` suelto, `PRAGMA ...` (solo SQLite, saltarlos en PG),
      `INSERT OR REPLACE/IGNORE` -> `ON CONFLICT`, `strftime(...)` -> equivalentes PG.
- IMPORTANTE: quitar (o hacer visible en un log) el `except: pass` que oculta
  errores de DDL, para que un CREATE que falle NO pase desapercibido nunca más.
